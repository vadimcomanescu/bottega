# Spec format

The default shape a spec takes, whether presented in the conversation or written into a ticket body. The parts:

- **What changes.** The behavior the work adds or alters, stated so the user can picture the result.
- **Acceptance criteria.** The observable conditions that decide the work is correct.
- **Definition of done.** What must be true to deliver beyond correctness: gates green, docs updated, evidence captured.
- **New or sharpened domain terms.** Terms the work introduces or redefines, in domain language, consistent with the affected `CONTEXT.md`.
- **Decisions.** Each choice Fable made on the user's behalf, its default flagged and written so the user can veto it in one read: the decision, the chosen default, one line of why.
- **Out of scope.** What the work deliberately excludes, so the boundary is explicit.
- **Visual prototypes.** Whenever the work has a visual surface, a wireframe of it. A wireframe looks like a wireframe: it shows layout and flow, never an image pretending to be the finished product.
