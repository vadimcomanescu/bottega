# A standalone spec lives on the work branch a run continues, not a parked spec branch

ADR 0004 kept spec content in the repo but chose an expensive transport for the standalone case: the spec sat on a permanent `bottega/spec-<slug>` branch, a parent issue linked it, child tickets carried per-unit scope, and the run that built the first ticket copied the file off the parked branch into its own. That machinery existed only to serve one spec fanning out into several parallel runs. That case is rare, and encoding it taxed the common case (one spec, one delivery) with a parked branch, a copy step, and a parent/child ticket structure a run had to walk to decide whether work was approved.

We serve the common case directly. A standalone spec session commits the spec on `bottega/<slug>`, the ordinary work branch, and files one issue that names it. A later run continues that branch: the spec file is already there, so there is no copy and no approval graph to walk. The run's PR merges the spec to trunk with the code that fulfils it. The branch is not permanent; it merges and is deleted with the run like any other. Prototype screenshots the spec embeds commit next to it on the same branch, so they reach trunk with the spec, which supersedes the permanent-spec-branch reason in ADR 0004 (the QA evidence branch in `0003-permanent-evidence-and-reader-contract.md` is a separate mechanism and is unchanged).

Specs still live in the repo and tickets still carry only queue state, the durable half of ADR 0004. What is dropped is the transport: permanent spec branches, the copy step, and parent-and-child tickets. Work too large for one PR is sliced inside the run, as it already is. If a spec ever genuinely needs delivery across several runs over time, merging its file to trunk by hand at that moment is a small deliberate act that needs no standing doctrine.
