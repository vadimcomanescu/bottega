# Specs live in the repo; tickets carry queue state; spec branches are permanent

A spec's content used to live in the tracker: the parent issue carried the spec and each child ticket carried its unit's slice (`0002-tickets-canonical-spec-branch-evidence.md`, now superseded on this point). That put the agreed record of what the code should do outside the repository: it never diffed with the code it describes, it was invisible to a session reading the repo, and later runs could not read past specs as grounding. The field's spec-driven tools (GitHub Spec Kit, Amazon Kiro, OpenSpec, and vendor guidance from Anthropic and others) all keep spec content in the repo for exactly those three reasons; the one argued exception (Steve Yegge's Beads) targets task-queue state, not spec content, because markdown task lists fork and contradict while a tracker carries claims and dependencies natively.

We adopt that split. The spec is a file, `docs/specs/<slug>.md`. A spec session commits it on the permanent `bottega/spec-<slug>` branch and the parent issue links it there; the run that delivers the first child ticket carries the file into its own branch, so the spec lands on trunk with the code it describes. A run-born spec is committed in the run worktree directly. Tickets keep only what the tracker is good at: the claim, the dependency edges, a scope statement, and the link to the spec. A work item carries the user's OK only when it is a child of a parent spec issue that links an agreed spec file; a plain tracker issue, however detailed, is input to a spec session, never an approved spec, which no surveyed agent platform treats it as either.

Spec branches are permanent for the same reason evidence branches are (`0003-permanent-evidence-and-reader-contract.md`): the prototype evidence the spec links lives on them, and deleting the branch kills those links retroactively. The cost is one small branch per spec session, cheaper than a spec whose evidence points at nothing.
