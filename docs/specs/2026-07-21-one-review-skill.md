# One review skill: the vendored reviewer's own document is the method

## Problem to solve

The operator asked for one thing, in these words: a review skill "that starts from byte identical autoreview skill" with the openclaw-specific material and Windows compatibility removed and a few of our rules bound in. What shipped instead re-authored the reviewer as a house document, deleted the vendored document it was meant to start from, and left a second skill restating the same review loop for open pull requests, plus two claim scripts guarding a concurrency problem no current agent system has. Today a reader cannot say which document governs a review, the vendored directory holds a reviewer with no instructions, and upstream updates can no longer land by re-copy.

## How we measure success

Over the runs that follow: every review runs under the vendored reviewer's document and no second review method is loaded anywhere; an upstream update lands by re-copying the directory and reconciling one diff; no session touches a claim script; the operator reads one document and knows exactly what "reviewed" guarantees.

## The launch post

Reviewing is one skill: autoreview, the vendored reviewer, carrying its author's method with bottega's few rules folded in where they belong. A run's integrated change is reviewed once by both model families in a single panel invocation, and the reviewers are blind: their prompt carries the reviewed repository's own review doctrine and the fixed standards baseline, never the spec or the plan. The maestro verifies every finding against the real code, accepts or refutes it, and dispatches the accepted findings to a fresh builder; the builder fixes, the project's tests and lint pass, and the reviewer runs again until nothing blocking remains. A separate check, isolated from the reviewers, reads the change against the agreed spec and reports what is missing, what nobody asked for, and what looks wrong against a quoted line of the agreement.

On an open pull request, the review threads are findings: each is verified, fixed or refuted with evidence, answered, and resolved. A clean review is recorded as one commit status on the reviewed head, naming the base it was judged against. Merging happens only after the checks are green and the merged head is exactly the reviewed one, and only when the owner armed the merge in their own words; a change touching authentication, money, permissions, stored data, or a destructive operation is never merged unattended.

Coordination uses what the tracker and the repository already provide: the issue's assignee, the work branch, and the pull request. No claim files, no marker trust rules, no lock protocol. The build phase runs one implementer per slice straight to the project's gates, and a simplification pass sweeps the landed slices before the integrated review so the review judges the code's final shape.

## Decisions

1. **The vendored document is the method, edited in place.** The reviewer's own instruction file returns to the vendored directory and is edited there: the openclaw-organization material (their issue-tracker fallbacks, bot-provenance rules, org path variants and variables) and the Windows sections are stripped; everything else stays as the author wrote it, including the parallel-test isolation path the operator uses. Precedent: vendoring with a local patch reconciled on sync, the standard vendor-branch practice. This reverses the previous spec's decision that a house skill absorbs the contract and the vendored prose leaves the load path; the evidence that changed the call is the operator's instruction, which was a literal text operation, and the observed loss of the author's method in the rewrite.
2. **One sync contract, pinned.** The vendored directory records the upstream revision it was copied from. An update re-copies scripts, tests, and license unchanged, then reconciles the document by applying the old-upstream-to-new-upstream diff to our copy. Precedent: three-way merge over vendored files, as version control has always done it.
3. **Accepted findings go to a fresh builder.** The reviewer's contract has the caller fix and rerun; in a run the maestro fulfils it by dispatching one fresh builder with the verified findings and the project's commands, nothing else, and the maestro never edits production code. The builder fixes, the tests and lint pass, and the reviewer runs again until no blockers remain; at two fix cycles without convergence the contract's own rule applies as the author wrote it, pause and reclassify every remaining finding before another edit, continuing while any is still an in-scope blocker. Precedent: the only explicit industry doctrine on who fixes flagged agent code requires an agent other than the author (Squad's independent-fix rule; Cursor's review autofix spawns a fresh agent); Devin leaves the question undocumented. This reverses this session's earlier resumed-builder direction; the evidence that changed the call is that a verified defect at a named line needs no authoring memory, and resume mechanics carried a recorded model-drift failure class.
4. **The landing skill is deleted.** Its three real facts fold into the reviewer's document: threads are findings, merge verification, and the armed merge with its risk paths. Everything else it carried restated the review loop or served the marker rules. Precedent: every surveyed agent product treats pull-request comments as the feedback loop and pushes fixes to the same branch; none ships a separate landing method.
5. **The claim scripts are deleted.** Ownership of work is the assignee, the branch, and the pull request. Precedent: no surveyed production system, platform, or team uses a lock or claim protocol; the platform's own multi-agent design chose no lock deliberately.
6. **The per-slice risk reviewer is removed.** A slice is one implementer, then the gates; the integrated panel is the one review. Precedent: the surveyed products that ship review at all ship one terminal review of the change. This reverses the 0.66.0 decision that risky slices get a blind reviewer; the evidence that changed the call is the operator's decision with the industry shape confirming it.
7. **The simplification pass returns to the build phase.** It runs over landed slices before the integrated review. This reverses the previous spec's decision that cleanup rides in the review's first round; the evidence that changed the call is the reviewer's own fixed prompt, which instructs its engines to report only defects and to skip cleanup, so a review invocation cannot carry that sweep.
8. **The panel's engine pins live in the reviewer's run addendum.** The routing skill keeps its exemption and points there. Precedent: one home per fact, this repository's own documentation rule.
9. **The skill keeps its vendored name.** The command surface is autoreview; the table lists it once. Renaming would blur the vendored identity the sync contract depends on.
10. **The previous spec's lesson survives.** Its rule, a vendored skill's contract binds the caller as written, is exactly what this change implements; its enforcement pointer moves to the new doctrine pins, which now also assert the vendored document is present and carries upstream's identity.
11. **The reviewers are blind; a separate check reads the change against the spec.** The review prompt carries standing rules only: the reviewed repository's root review doctrine and the fixed standards baseline, never the spec or the plan. Spec conformance is its own pass, isolated from the reviewers: one fresh agent from the other model family reads the change and the agreed spec and reports three things, each quoting the line of the agreement it rests on: what was agreed but is missing or partial, what was built that nobody asked for, and what looks wrong against the agreement. With no spec available it says so instead of guessing. Neither pass sees the other's findings, and nothing reranks across them. Precedent: every shipped reviewer judges the diff against standing rules only; the two-axis separation is the published method of the one review skill that checks conformance at all; and the authoritative conformance check remains driving the product's scenarios at QA, which is what the one company with a no-human-review policy ships.
12. **A defect the review missed becomes a standing rule.** When QA or an incident catches what the review did not, the run files the lesson and adds a rule to the reviewed repository's review doctrine, stating the unsafe behavior, why it matters, and the safe path, placed near the code it governs; deterministic checks go to the project's tooling, never into review rules. Precedent: the one team that removed humans from its review loop earned it by feeding every miss back into the reviewer's evaluation set, and the rule-writing discipline follows the other model vendor's published review-rules guidance.

## Details

The consumer sweep touches every surface the inventory found: the readme's command table, pipeline story, design-decision paragraph, and roles table; the repository map and its scripts row; the review-guidelines pointer that exempts the review gate's engine pins; the routing skill's two review lines; the maestro's build, review, and issue-handling sentences; the closer's status and post-open references; the setup skill's claim-script and label reconciliation; the plugin manifests; the runtime symlink directories; and the doctrine tests, where the pins that read the deleted skill are replaced by pins on the vendored document's presence and identity, the run addendum's engine pins, the fresh-builder fix dispatch, and the maestro interlock. The claim-script test suites are deleted with the scripts. The superseded spec file remains in the repository as the record of the decision this one reverses.

## Acceptance criteria

- The vendored directory carries the reviewer's document, license, scripts, tests, and the pinned sync note; the document's method text is upstream's at the pinned revision except the named strips and the run addendum.
- The review skill directory, the landing skill directory, both claim scripts, and both claim test suites do not exist, and no method text, readme line, manifest entry, or symlink references them.
- The maestro's build phase has one implementer per slice, the gates, and the simplification pass, and no per-slice reviewer; its review phase invokes the vendored reviewer blind and dispatches the spec-conformance check.
- The full verification gate is green, with the doctrine pins asserting the new direction.
- The release gate is satisfied: version bumped above the current base, readme table synced, release title in the required form.

## Out of scope

The routing table and model choices. The panel skill. The spec method. The QA method. Any auto-merge or review-less mechanism. Adopting draft pull requests as a work-in-progress signal.

## Deferred to the build

The exact wording of every edited sentence. The upstream revision value the sync note pins. The itemized list of stripped upstream sections, recorded in the pull request. The choice of load-carrying sentences the doctrine tests pin.
